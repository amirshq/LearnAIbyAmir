🗣 NLP / Intent Recognition
🎯Intent
A named goal or action the user wants to accomplish. The top-level classification of what the user is trying to do.
"BookFlight", "CheckBalance", "CancelOrder"
 Intent
Core unit; every bot is a collection of intents.
 Topic
A topic contains trigger phrases and the conversation path for one user goal.
 Intent
Same concept; matched via ML classifier or exact phrases.
 Tool / Chain
No formal "intent"; routing is done by the LLM selecting a tool or chain to invoke.
 —
No explicit intent layer; the LLM interprets user messages directly and routes to agents.
 Task
Closest analog; a task defines what an agent should accomplish.
💬Utterances / Training Phrases
Sample phrases users might say to trigger an intent. Used to train the NLU classifier.
"I want a pizza", "Order me a large margherita", "Get me food"
 Utterances
Listed per intent; Lex uses them to train its ML model.
 Trigger Phrases
Phrases added to a topic that cause it to activate.
 Training Phrases
Added per intent; annotated with entity tags for extraction.
 Prompt / System Prompt
No training phrases; the LLM itself handles NLU. Routing is instruction-based.
 —
Not applicable; LLM handles all NLU natively from the conversation.
 Task Description
Natural language description of what the task is; the LLM interprets it directly.
🏷Entity / Slot
A typed variable extracted from user input — a piece of information needed to fulfill the intent.
City = "Montreal", Date = "tomorrow", Size = "large"
 Slot
Typed data points Lex extracts. Each intent has a slot list.
 Variable / Entity
Variables store extracted values; entities define the type (e.g. city, date).
 Entity / Parameter
Entities define types; parameters are per-intent instances of those types.
 Tool Input Schema (Pydantic)
Tool arguments defined via Pydantic BaseModel. LLM fills them from conversation.
 Function Parameter
Tool/function call arguments extracted by the LLM when invoking a registered function.
 Task Input / Context
Passed as structured context when a task is created. No formal slot-filling loop.
🔁Slot Filling / Elicitation
The multi-turn loop where the bot asks follow-up questions to collect all required slots before fulfillment.
Bot: "What city?" → User: "Paris" → Bot: "What date?"
 Slot Elicitation
Built-in; Lex auto-prompts for missing slots using configured prompts.
 Question Nodes
Explicit Ask-a-Question nodes in the topic canvas collect variable values.
 Form / Parameter Filling
CX Form nodes collect all parameters before moving to the next page.
 Multi-turn Chain / Human-in-the-loop
Managed manually or via LangGraph interrupt nodes with state checkpoints.
 Multi-agent Turn / Human Proxy
UserProxyAgent simulates a user turn; missing info is requested via follow-up messages.
 —
No built-in slot filling; handled via custom agent logic or human input callbacks.
🎲Confidence Score / Fallback
A score indicating how certain the NLU model is about its intent match. Below a threshold, a fallback/default intent fires.
Confidence: 0.43 → triggers FallbackIntent
 AMAZON.FallbackIntent
Special built-in intent fired when no other intent meets the confidence threshold.
 Fallback Topic / Escalate
"Fallback" system topic; can escalate to human agent.
 Default Fallback Intent
Fires when NLU confidence is below threshold; configurable per flow.
 Default Tool / Catch-all Prompt
No native concept; handled by prompt design or a default routing branch.
 —
LLM responds naturally even to ambiguous input; no explicit fallback mechanism.
 —
Not applicable; task routing is LLM-driven.
🤖 Agent / Execution
🤖Agent / Bot
The top-level autonomous entity that perceives input, reasons, and takes actions or generates responses.
CustomerSupportBot, ResearchAgent, SalesAgent
 Bot
The Lex bot; contains all intents, slot types, and locales.
 Copilot / Agent
The published Copilot; wraps all topics and knowledge sources.
 Agent
A Dialogflow CX agent; contains all flows, intents, and entity types.
 Agent
AgentExecutor or LangGraph graph node that reasons + calls tools in a loop.
 Agent (AssistantAgent)
ConversableAgent subclass. AssistantAgent is the LLM-powered agent role.
 Agent
Defined with role, goal, backstory, and tools. The atomic worker unit.
🔧Tool / Action / Function
A callable function the agent can invoke to interact with the outside world — APIs, DBs, code executors, etc.
search_web(), query_db(), send_email(), run_code()
 Lambda Hook (Fulfillment)
AWS Lambda function triggered on slot capture or intent fulfillment.
 Action / Power Automate Flow
HTTP calls or Power Automate flows triggered within a topic.
 Webhook / Fulfillment
HTTP webhook called when an intent or page fires; returns dynamic responses.
 Tool (@tool decorator)
Python function wrapped with @tool. LLM selects and calls it via function calling.
 Tool / Registered Function
Functions registered on agents; LLM invokes via OpenAI-style function calling.
 Tool
BaseTool subclass or @tool-decorated function assigned to an agent.
👤Human Proxy / User Simulator
A component that stands in for a human user in automated or multi-agent pipelines — handles input/output on behalf of a person.
Testing pipelines, auto-feedback loops, HITL approval
 —
Not applicable; Lex is always user-facing directly.
 —
Not applicable for single-bot architecture.
 —
Not applicable; Dialogflow is user-facing.
 Human-in-the-loop (interrupt)
LangGraph interrupt() pauses the graph and waits for human input before resuming.
 UserProxyAgent
Core AutoGen agent that represents the human. Can auto-reply or request real input.
 Human Input (human_input_mode)
human_input_mode="ALWAYS" on an agent pauses for real human approval.
📋Role / Persona
The identity, expertise, and behavioral framing given to an agent that shapes how it responds and what it focuses on.
"You are a senior financial analyst…", role="Researcher"
 —
Not a concept in Lex; behavior defined by intents and Lambda logic.
 Identity / Instructions
Set in "Configure" tab; defines the copilot's name, tone, and scope.
 Agent Description
High-level description used by generative playbooks.
 System Prompt
Injected into every LLM call to define persona and behavioral constraints.
 system_message
Constructor arg on ConversableAgent that defines the agent's role and behavior.
 role + backstory
Explicit fields on Agent: role (short title), backstory (detailed persona narrative).
⚡LLM / Model Config
Configuration of which underlying language model the agent uses and its inference parameters (temperature, max_tokens, etc.).
model="gpt-4o", temperature=0.2, max_tokens=1024
 —
Lex uses its own built-in NLU; no LLM config exposed directly.
 Generative AI Model Settings
Configured in "AI capabilities"; selects Azure OpenAI model and controls.
 Generative Model Settings
Per-agent or per-playbook model selection (Gemini family).
 LLM / ChatModel object
ChatOpenAI(model=..., temperature=...) or any LangChain-compatible LLM.
 llm_config dict
Passed to agent constructor: {"model": "gpt-4o", "temperature": 0.1, ...}
 llm parameter on Agent
Agent(llm=ChatOpenAI(...)) or a string model name; supports LiteLLM backends.
🧠 Memory / State Management
🗃Session / Context
The container that holds all data for one user conversation — variables, history, and state — from start to end.
session_id = "user_789_20240312"
 Session
Session attributes (key-value map) persist for the session lifetime.
 Conversation Session
Scoped variables (conversation vs. global) stored per session.
 Session
Session parameters carry values across pages within one conversation.
 State (TypedDict)
In LangGraph, the entire graph state dict is the session. Passed node-to-node.
 Chat History (messages list)
The running list of messages exchanged between agents is the session state.
 Context / shared_memory
Task output is passed as context to subsequent tasks; crew-level shared memory optional.
📜Conversation History / Memory
The record of past messages and interactions that the agent uses to maintain coherent multi-turn dialogue.
ChatHistory, message buffer, summarized past turns
 Session Attributes
No native message history; must be managed in Lambda or session attributes.
 Conversation Variables
Variable storage per conversation; full history not exposed natively.
 Session Parameters
Values persist within a session; full history managed externally.
 Memory / ChatMessageHistory
ConversationBufferMemory, ConversationSummaryMemory, or LangGraph state messages list.
 messages (chat history)
The full message list on each agent; can be cleared or summarized to manage token budget.
 memory=True on Crew
Enables short-term (in-task), long-term (SQLite), and entity memory via embeddings.
💾Long-term / Persistent Memory
Storage that survives beyond a single session — user preferences, historical facts, past interactions stored in a DB.
User preference store, vector DB recall, user profile
 DynamoDB (external)
Must be built manually via Lambda + DynamoDB; not native to Lex.
 Global Variables / Dataverse
Global-scope variables or Dataverse/Dynamics 365 for persistent user data.
 External DB via Webhook
Must be implemented in fulfillment webhook; Dialogflow doesn't persist cross-session natively.
 VectorStore / LangMem
ChromaDB, Pinecone, etc. via LangChain retrievers. LangMem adds background memory updates.
 External store + TeachableAgent
TeachableAgent writes facts to a memo store (vector DB) and retrieves on future sessions.
 Long-term Memory (SQLite + embeddings)
Stored in ~/.crewai/memory/; crew recalls relevant past runs using semantic search.
🔀 Flow / Orchestration
🗺Flow / Graph / Pipeline
The top-level structure that defines how conversation steps or agent actions are sequenced and connected.
Start → Greet → CollectInfo → Confirm → Fulfill → End
 Intent + Slot Flow
Implicit flow: Lex follows slot elicitation order then calls fulfillment Lambda.
 Topic Canvas
Visual node-based canvas showing the entire conversation path for a topic.
 Flow + Pages
CX introduces Flows (sub-graphs) each containing Pages (states).
 LangGraph Graph (StateGraph)
StateGraph with nodes (functions) and edges (transitions). Supports cycles and branching.
 GroupChat / Swarm
GroupChatManager routes messages between agents. Swarm is a newer topology.
 Crew (process="sequential"/"hierarchical")
The Crew orchestrates agent order. Sequential = chain; Hierarchical = manager routes.
📍State / Node / Page
A discrete point in the conversation or execution where a specific action happens — a question is asked, a decision is made, or data is processed.
AskCityNode, ConfirmOrderPage, ExtractSlotStep
 Dialog Step (implicit)
Lex manages state implicitly through slot collection; no explicit node concept.
 Node (Message / Question / Action)
Canvas nodes: Message, Question, Condition, Action, End Conversation.
 Page
CX core unit; each Page has entry fulfillment, a form, routes, and event handlers.
 Node (function in graph)
Any Python function added to the StateGraph is a node. Receives and returns state dict.
 Agent turn / message handler
Each agent's reply() constitutes a step; no explicit "node" abstraction.
 Task
Each Task is a discrete step assigned to an agent; tasks are the nodes of the crew pipeline.
➡Transition / Edge / Route
A conditional or unconditional link that moves execution from one state/node to the next based on conditions or LLM decision.
if slot_filled → next_page, if condition → branch_A else branch_B
 Conditional Branch (Lambda)
Branching is done in Lambda return value (dialogAction); no visual routing.
 Condition Node / Branch
Condition nodes create if/else branches based on variable values.
 Route / Transition Route
Condition-based or intent-triggered routes move from page to page.
 Edge / Conditional Edge
add_edge() for fixed; add_conditional_edges() for dynamic routing based on state.
 Handoff / speaker_selection_func
GroupChatManager selects the next speaker; custom functions enable conditional routing.
 Task dependency (context=)
context=[prev_task] chains tasks. Hierarchical process uses manager LLM to route.
🔄Loop / ReAct Cycle
The think→act→observe loop where the agent repeatedly reasons about what to do next, calls a tool, observes the result, and continues until done.
Reason → Call search_tool → Observe result → Reason again → Answer
 —
Not applicable; Lex is rule/slot-driven, not reasoning-loop-driven.
 Generative Actions (Orchestrator)
Autonomous mode uses an AI orchestrator to decide which action to call next.
 Playbook (Generator Loop)
CX Playbooks enable a generator loop for autonomous multi-step task completion.
 AgentExecutor / LangGraph cycle
AgentExecutor runs the ReAct loop. LangGraph uses explicit cycles in the graph (should_continue edge).
 ConversableAgent reply loop
Agents exchange messages in a loop; each exchange is one think-act step.
 Agent execution loop
Under the hood, each CrewAI agent uses a ReAct-style loop when executing a task.
🏁Fulfillment / Final Answer
The step where the agent delivers the final response or performs the concluding action after all information is gathered and processing is done.
"Your flight to Paris on Jan 15 is booked! Confirmation: XY291"
 Fulfillment Lambda
Lambda function called after all slots are filled; returns final response to user.
 Message Node (End)
Final message or "End Conversation" node delivers the closing response.
 End Flow / Response Fulfillment
Webhook response or static text response on the final page transition.
 Final Response / END node
When the graph reaches END, the last state value (typically "messages[-1]") is the answer.
 Termination (is_termination_msg)
Conversation ends when termination condition is met; last assistant message is the output.
 Task Output / Crew Output
Each task returns a TaskOutput; the final task's output is the crew's result.
📡Handoff / Delegation
The transfer of control from one agent to another — either escalating to a specialist or handing off to a human.
BillingAgent → EscalateToHuman, ResearchAgent → WriterAgent
 —
Not native; can forward session to Connect agent queue via Lambda.
 Escalate to Human / Transfer Topic
Built-in "Transfer Conversation" node hands off to live agent queue.
 Live Agent Handoff
Built-in event that routes conversation to CCAI-integrated human agent.
 Handoff (LangGraph node)
A conditional edge routes to a different subgraph or agent node based on state.
 Agent-to-Agent Message / Handoff
An agent sends a message to another agent; GroupChatManager facilitates multi-agent handoffs.
 delegate_work Tool
Agents can call delegate_work() to hand a sub-task to a colleague agent by name.
